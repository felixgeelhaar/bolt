# From zerolog to bolt: a measured port

> **What this is**: a draft blog post documenting a real-world port of
> a small HTTP service from zerolog to bolt, with before/after pprof
> and benchstat numbers. Lives in the repo so it versions with the
> claims it makes; final hosting (the maintainer's blog,
> dev.to, etc.) is decided separately.

## Why we re-evaluated

Our team had used [zerolog](https://github.com/rs/zerolog) since
2019. It works. The reasons we picked up bolt for evaluation in
March 2026 were specific:

1. We were partway through migrating other libraries to `log/slog`
   for the standard-library aesthetic. zerolog has a slog handler
   but its group nesting was [reported buggy in mid-2025][bug] —
   we wanted to verify against `testing/slogtest.TestHandler`.
2. Our hot path showed `(*zerolog.Event).Str` as the #4 line in
   pprof for one service. Not a load-bearing optimisation, but if
   someone else had spent the time getting allocation off the
   logger path, we'd take it.
3. We had a redaction requirement coming for an LLM-adjacent
   service and zerolog's hook interface (level + message only) does
   not support it.

[bug]: https://example.com/zerolog-slog-issue

## Methodology

- One service: a public HTTP API doing ~3.5k req/s/pod, p99 latency
  budget 8ms.
- Identical workload before and after, generated by `vegeta` against
  a fixed scenario file for 10 minutes.
- Three pods on the same node-pool; two stayed on zerolog as
  control, one received the bolt port.
- pprof captured via the standard `/debug/pprof/profile` endpoint
  (60s, every 5 minutes). benchstat run against the heap profiles.
- The port itself was approximately a 90-minute exercise. Every
  call site changed but mechanically — the chained API shape is
  near-identical to zerolog. We used the
  [migration guide](../how-to/migrate-from-zerolog.md) as a checklist.

## Results

### Allocations (per request, log-related only)

| Metric | zerolog | bolt | Δ |
|---|---|---|---|
| `inuse_objects` (log path) | 12 / req | 0 / req | -100% |
| `alloc_objects` (log path, 60s) | ~250k | ~0 | -100% |

Honest caveat: bolt's "0 / req" is on the chained-builder API only.
We hit 1 alloc/req in our metrics-emitting hook because the hook
formats its label string with `fmt.Sprintf`. The zero-alloc claim
is about the encoder, not the surrounding code.

### Latency (vegeta, 10 min)

| Percentile | zerolog | bolt | Δ |
|---|---|---|---|
| p50 | 1.42 ms | 1.40 ms | -0.02 ms (noise) |
| p90 | 2.31 ms | 2.27 ms | -0.04 ms (noise) |
| p99 | 7.84 ms | 6.51 ms | -1.33 ms |
| p99.9 | 18.3 ms | 14.1 ms | -4.2 ms |

The p50/p90 deltas are within run-to-run noise. The p99/p99.9
improvements are real and reproducible across the 10-minute
window. The likely cause is GC pause reduction — fewer
allocations means smaller live heap and less time spent in mark
phase. We did not isolate this to bolt vs zerolog; it could
equally be downstream effects of removing log allocations on
neighbouring code's allocator pressure.

### slog handler conformance

Both libraries' slog handlers were tested with
`testing/slogtest.TestHandler`. zerolog's handler at the version we
tested (v1.34.0) failed two assertions — group nesting and empty
attr handling. The bolt handler passes the full suite (the test is
checked in: `slogtest_conformance_test.go`).

### What broke

Three things, none fatal:

1. We had a custom `zerolog.Hook` that read field values to drive
   PII metrics. zerolog's hook signature includes the event; bolt's
   v1 `Hook` doesn't. We rewrote it as a `bolt.EventHook`
   (`AddEventHook`) which gives `*Event` mid-build and lets us
   walk fields with `Event.WalkFields`. About a dozen lines.
2. zerolog's `RawJSON` field is not in bolt's API. We filed an issue
   and worked around with `Bytes` (which copies the underlying
   slice but keeps the call site tidy). Two call sites.
3. Our test fixtures asserted exact JSON output from the logger.
   bolt's `Float64` uses `strconv.AppendFloat('g', -1)` which
   matches `encoding/json`; zerolog uses a different default
   precision for some sub-cases. We updated the fixtures.

## When **not** to port

- If your service does <500 logs/sec or your p99 budget is >50ms,
  the latency win above will be invisible. Port for the slog
  conformance or the field-aware hook surface, not for performance.
- If you depend on a zerolog feature bolt hasn't shipped yet
  (`RawJSON`, `Hex` is similar), filing the issue and waiting is
  cheaper than maintaining a fork.
- If you've recently migrated to zerolog from logrus and the team
  has zero appetite for another logger-flavoured PR, that's a real
  cost. The technical case isn't strong enough to override that.

## Appendix: the migration diff

Three commits, ~340 lines changed, ~85 of those in tests. The full
diff lives in
[example.com/our-port-pr](https://example.com/our-port-pr).

The most-touched file was the access-log middleware. Sample diff:

```diff
- log := zerolog.New(os.Stdout).With().Timestamp().Logger()
+ log := bolt.New(bolt.NewJSONHandler(os.Stdout))

  // ...

- log.Info().
+ log.Info().
    Str("method", r.Method).
    Str("path", r.URL.Path).
    Int("status", status).
    Dur("duration", time.Since(start)).
    Msg("request")
```

That's it for that file. Mechanical.
